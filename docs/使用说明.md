# Poi 图像识别自动化框架使用说明

这是一个运行在 Windows 上的 Poi / 舰娘 Collection 自动化脚本框架。首版目标是搭好可扩展骨架：识别 Poi 浏览器左半侧 `1200x720` 游戏画面，通过手写 YAML 规则执行截图、状态判断和鼠标点击。

本项目不使用 AI，不接入自动学习。所有入口、按钮、阵型、血量判断都通过配置和模板手动维护。

## 安装

建议使用 Python 3.11 或更新版本。

```powershell
cd E:\Codex_project\Poi图像识别自动化
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -e .
```

启动 GUI：

```powershell
poi-auto
```

也可以直接运行：

```powershell
python -m poi_auto.app.main
```

## 基本配置

全局配置在 `config/default.yaml`。

关键字段：

```yaml
window:
  title_keyword: "Poi"

game:
  logical_width: 1200
  logical_height: 720
  crop_mode: "left_half"
```

程序会查找标题包含 `title_keyword` 的窗口，取该窗口客户区左半侧作为游戏画面。如果实际左半侧不是 `1200x720`，截图会缩放到逻辑分辨率后再做识别；点击时会从逻辑坐标换算回真实屏幕坐标。

## GUI 功能

主窗口包含：

- 刷新截图：预览 Poi 左半侧游戏画面。
- 启动出击：后台循环执行出击状态机。
- 停止：请求任务停止，当前动作结束后不会继续点击。
- 单步执行：只执行出击状态机的第一步，适合调试点击点。
- 配置摘要：展示当前窗口、分辨率和出击配置。
- 运行日志：显示截图区域、缩放比例、状态、点击坐标和错误。

## 出击规则

出击规则在 `config/tasks/sortie.yaml`。

状态结构示例：

```yaml
home:
  detect:
    type: "always"
  actions:
    - type: "click"
      point: "sortie_entry"
    - type: "wait"
      ms: 800
  next: "map_select"
  timeout_ms: 5000
  on_fail: "stop"
```

支持的识别条件：

- `always`：总是匹配，用于先跑通坐标点击流程。
- `template`：使用 OpenCV 模板匹配。
- `manual_flag`：预留给 GUI 或调试开关。

支持的动作：

- `log`：写日志。
- `click`：点击 `points` 中配置的逻辑坐标。
- `wait`：等待指定毫秒。
- `formation`：点击 `formation_points` 中当前配置阵型。
- `damage_check`：执行血量规则预留逻辑。
- `stop`：停止当前任务。

## 坐标规则

所有坐标都按游戏逻辑分辨率 `1200x720` 填写。

示例：

```yaml
points:
  sortie_entry: { x: 1040, y: 520, name: "出击入口示例" }
```

这些坐标目前是占位示例。正式使用前，需要用截图预览和单步执行确认真实坐标。

## 模板目录

模板图放在 `assets/templates/`。

例如：

```text
assets/templates/sortie/damage/heavy.png
assets/templates/sortie/damage/medium.png
```

规则中这样引用：

```yaml
detect:
  type: "template"
  template: "sortie/damage/heavy.png"
  threshold: 0.88
```

模板缺失时程序不会崩溃，会在日志中提示缺失路径。

## 后续扩展

新增任务建议按下面方式组织：

```text
src/poi_auto/tasks/repair/
src/poi_auto/tasks/resupply/
src/poi_auto/tasks/expedition/
config/tasks/repair.yaml
config/tasks/resupply.yaml
config/tasks/expedition.yaml
```

约定：

- `device/` 只处理窗口、截图和点击。
- `vision/` 只处理模板匹配和识别。
- `tasks/` 只处理游戏流程。
- `gui/` 只处理界面展示和按钮交互。
- `core/` 只处理配置、运行上下文和任务线程。

这样可以逐步加入远征、修理、补给、任务领取等功能，而不会把所有逻辑堆进一个脚本。
